## Description

Close M13 against the real product bar: a developer laptop must be able to receive and place real PSTN calls through the new telephony control plane, not just validate wiring in k3d. This task defines and implements a supported local-real-call profile where LiveKit Server plus `livekit-sip` run inside the local k3d cluster, while the `k3d` load balancer publishes the real SIP/RTP host ports on the laptop’s public IP/FQDN. `platform-api`, workers, DB, and admin UI remain local. It also adds the smallest operator-facing outbound test-call path that can start `VoiceCallWorkflow` against a real E.164 number using the new telephony inventory.

## Implementation Notes

- This task is intentionally not a k3d-only wiring exercise. Real PSTN proof still requires a public SIP/media edge, but the supported local profile now publishes that edge through the laptop-hosted `k3d` load balancer instead of a separate host-level LiveKit process.
- Keep `platform-api`, Temporal workers, DB, and admin UI local. Do not “solve” this by moving the whole platform to a public cluster.
- Reuse the new telephony inventory and governed agent model. Do not route outbound proof through the legacy browser voice path or campaign-specific workflows unless the generic operator test-call path is impossible.
- Treat inbound proof and outbound proof as separate acceptance gates. One without the other does not close the milestone.
- Capture the exact public-IP/FQDN, SIP/RTP/TURN expectations in the runbook so this becomes a repeatable developer workflow instead of folklore.

The local NGINX ingress was forcing `Connection: keep-alive` through `nginx.ingress.kubernetes.io/connection-proxy-header`, so valid `/rtc` WebSocket handshakes reached LiveKit as non-upgraded HTTP requests and returned `404`. `infrastructure/kubernetes/base/platform/core/livekit-ingress/ingress-livekit.yaml` must preserve WebSocket upgrade headers for `livekit-livekit-server`; the regression is locked by `tests/architecture/test_k8s_livekit_package.py` and cluster-backed E2E coverage in `packages/platform-core/tests/e2e/test_call_ops_lifecycle_compose.py` for both draft and published browser-test signal connections.
- The next browser workbench failure was a deeper WebRTC media-path bug, not another auth or signal bug. `/rtc` websocket checks passed, LiveKit created the room, the agent joined, and agent audio was published, but the browser participant closed with `SIGNAL_SOURCE_CLOSE` and ICE stats showed zero responses. The local `k3d` load balancer published HTTP and SIP/RTP, but not LiveKit Server's browser RTC TCP mux on `7881/TCP`. `tools/scripts/infra/k3d-up.sh` now publishes `7881/TCP` for voice-enabled local clusters, prints it in the startup output, and treats older clusters missing that mapping as stale. The browser-session E2E now uses LiveKit SDK `Room.connect` and waits for the agent participant instead of only opening the signal websocket.
- GCP production voice cutover is no longer allowed to be hidden behind an app-only overlay. The GCP production overlay now renders `agent-worker` with the Terraform-owned `agent-worker` Kubernetes service account, no mounted JSON ADC, the GAR `agent-worker` image pin, and scoped `agent-worker-runtime-secrets`. The shared local LiveKit ingress was moved out of `base/platform/core` so cloud-mode GCP production does not need a delete patch or accidentally render `livekit.grove.localtest.me`; local/CI and Hetzner self-hosted paths opt into that ingress explicitly.
- Test coverage improved but is still not enough to claim this class is closed. We now have regression coverage that local `tools/scripts/k8s-runtime-secrets.sh` rewrites `LIVEKIT_URL` / `LIVEKIT_API_URL` for the local cluster, and architecture coverage that the `agent-worker` image ships the clinic plugin plus does not suppress the initial greeting. What is still missing is a deterministic local-real-call runtime test that proves `gcp-adc` exists and that `agent-worker` sees `GOOGLE_APPLICATION_CREDENTIALS` / `GOOGLE_CLOUD_PROJECT` before a real handset call is placed. Until that lands, `tools/scripts/telephony/real_call_precheck.py` remains the operational stop sign instead of a full automated proof.
- Real PSTN acceptance proof is still open overall. This task is not done until the inbound laptop-hosted public-IP runbook has also been executed against a real carrier path.
