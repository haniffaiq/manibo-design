## Goal

Take the pain discovered after M26 from tribal knowledge into a tracked follow-on milestone: reviewed bot PRs must recover automatically when current-head gate state is missing, GitHub PR truth must be parsed once instead of differently in every agent, review/follow-up logic must stop depending on comment archaeology, and fingerprint/backlog ownership must stay boring instead of spawning duplicate stale bot PRs.

## Current Status Snapshot

1. The shared tracked-PR contract now lives in `tools/agents/tracked_pr_state.py` and is consumed by orchestrator, follow-up, mergeability, and review bot.
2. Follow-up policy is now explicit and typed in `tools/agents/pr_followup_policy.py`, and run artifacts expose decision records even when `processed_prs=0`.
3. Reviewed bot PRs with missing current-head gate state now have two cooldown-bounded rescue paths: rerequest the newest stale zero-run current-head check suite when GitHub already has one, or allow follow-up to push one bounded empty refresh commit after the PR stays idle past the stale window. If the refreshed head stays stale again, it becomes eligible for another rescue cycle after the same cooldown window instead of parking forever.
4. Root-cause fingerprint marker rendering/injection is now shared across issue upsert, issue execution, PR creation, and PR supersedence logic.
5. Existing heavy-runner hosts now reconcile the canonical prewarm and cleanup host assets from `cloud-init.yml.tftpl` during bootstrap and heavy-host rollout, so old first-boot copies no longer survive forever on a repaired VM.
6. `Monitor - CI Scheduler Health` now measures stale zero-run suites from latest suite activity (`updated_at` after rerequest, falling back to `created_at`), so the 30-second post-rerequest recheck only alerts when GitHub still shows no new activity.
7. Production release run `24311654953` exposed one remaining heavy-lane gap: `k3d cluster create --wait` can wedge after CoreDNS alias injection without an inner repo-side detector, so M26.1 now adds a bounded create watchdog and diagnostics on the cluster-create path.
8. Production runs `24374812828`, `24385438368`, and runner-health run `24388346212` exposed one more heavy-host gap: the single heavy runner still depends on idle-only cleanup to recover from disk accumulation, and the k3d node container still inherits the default soft `nofile=1024`, so M26.1 now adds repo-owned heavy self-heal plus explicit k3d runtime ulimits.

## Design Decisions

1. **Tracked PR state becomes one shared typed contract**. Gate status, current-head checks, authoritative review state, mergeability, unresolved-thread state, and normalized root-cause fingerprint must be parsed once and reused.
2. **Workflow replay must preserve PR-path truth**. Maintenance workflows can still run on `main`, but any bot-PR gate rescue must carry the PR head ref plus enough PR/base/head metadata to recreate the real PR-path classification instead of faking a `workflow_dispatch` run on `main`.
3. **Authoritative review state owns machine decisions**. Follow-up and mergeability should trust repo-owned current-head review state first and use comment-marker fallback only as a compatibility bridge.
4. **Follow-up decisions must be explicit and observable**. Replace the stringly `_select_action()` blob with typed decisions and reason codes so `processed_prs=0` is diagnosable instead of mysterious.
5. **Fingerprint ownership is one shared contract**. Parsing, normalization, family matching, and marker injection must come from one helper module consumed by issue upsert, issue execution, PR creation, and PR follow-up.
6. **M26.1 is a refactor/reliability milestone, not a feature land grab**. No new runner choreography, no new review authorities, no new branch-protection surfaces, and no feature work hidden inside control-plane cleanup.

## Acceptance Criteria

- [x] A reviewed bot PR with missing current-head `Merge gate` status can trigger cooldown-bounded targeted gate-refresh retries that preserve PR-path classification and metadata instead of falling back to a `main`-only `workflow_dispatch` replay.
- [x] Orchestrator, follow-up, mergeability, and review bot consume one shared tracked-PR state contract instead of re-parsing GitHub payloads independently.
- [x] Follow-up can decide current-head review readiness from authoritative review state without requiring a top-level blocking summary comment.
- [x] Follow-up action records expose explicit blocked/awaiting/rerun/merge reasons so `processed_prs=0` is diagnosable from artifacts.
- [x] Root-cause fingerprint parsing/normalization/injection is shared across issue upsert, issue execution, PR creation, and PR follow-up.
- [x] Existing heavy-runner hosts pick up the current prewarm host scripts and units during bootstrap instead of depending on first-boot `cloud-init` state forever.
- [x] Scheduler-health zero-run detection ages suites from latest suite activity so a successful rerequest gets a real cooldown window before Slack and issue escalation.
- [x] `wiki/architecture/ci.md` and this milestone/task set describe the final control-plane contract without contradicting runtime behavior.
- [x] The heavy `k3d` bootstrap path fails a wedged cluster-create attempt within a bounded inner timeout and emits targeted diagnostics instead of waiting for outer cancellation.
- [x] The heavy runner attempts repo-owned repair before reporting reclaimable disk pressure as a hard failure, and the k3d create path passes an explicit high `nofile` runtime ulimit.
