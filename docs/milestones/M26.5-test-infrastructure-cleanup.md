## Problem

### 1. Shell script sprawl (87 shell + 63 Python scripts in one flat directory)

`tools/scripts/` is a 150-file flat directory mixing CI plumbing, dev tools, test runners, observability queries, infra setup, and admin scripts. You can't tell who a script is for or when to run it.

### 2. Scripts that shouldn't exist

Shell test-runner scripts are only justified when orchestration is needed: starting infra, port-forwarding, setting env vars, managing lifecycle. For everything else, `uv run pytest <path>` or `pnpm test` is the command — a wrapper script adds nothing.

**Scripts that need orchestration (keep):** k3d E2E (port-forward, OIDC mock, env mutation), compose E2E (up/down lifecycle), real-provider evals (credential gates), voice tests (LiveKit env), web E2E (Playwright, built server, solutions env), traceability (multi-mode orchestration).

**Scripts that are just `uv run pytest <one-file>` (kill):** `run_clinic_booking_e2e.sh`, `run_clinic_handoff_e2e.sh` — these are Makefile targets, not scripts.

## Principle: when does a test need a shell script?

**YES — needs a script** (infra orchestration):
- k3d E2E: port-forwarding, OIDC mock, deployment env mutation, cleanup
- Compose E2E: compose up/down lifecycle, env export
- Real-provider evals: credential gates (check `GOOGLE_APPLICATION_CREDENTIALS` etc.)
- Voice E2E: 12+ LiveKit env vars, room lifecycle
- Web E2E: Playwright server build, solutions env, auth mode switching
- Traceability: multi-mode (compose/k3d/external), canary orchestration

**NO — just a command** (no script needed):
- Unit tests: `uv run pytest <path>` — Makefile target or CI step
- Integration tests: `uv run pytest <path>` — same
- Architecture tests: `uv run pytest tests/architecture/` — same
- Web unit tests: `pnpm test` — same
- Linting: `uv run ruff check .` / `uv run pyright` — same

## Non-goals

- Changing test logic or assertions
- Adding new tests
- Changing CI workflow topology
- Refactoring large script internals (k3d-test-e2e.sh, run_traceability_harness.sh)
- Reorganizing Python scripts in `tools/scripts/*.py` (tracked in M26.6)
