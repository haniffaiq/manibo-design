## Goal

Wire the reused Affidea prompts, KB artifacts, Affidea tools, and grouped-flow
runtime into a working NFQ voice-booking agent profile. This milestone proves
the caller-facing flow locally through browser voice and platform runtime seams,
without yet claiming production call quality.

## Design Decisions

1. The runtime agent is a solution starter/profile owned by
   `solutions/appointment_booking`, not a second demo app.
2. The grouped authoring shape is front desk + consultation + MRI + auth. If a
   resolved flat config is still needed for compatibility, it is generated from
   that grouped source rather than hand-maintained separately.
3. Browser rehearsal and PSTN use the same voice agent profile. Transport
   differences stay in LiveKit/runtime config, not prompt logic.
4. Human handoff is a normal tool/action path that signals the durable call
   lifecycle. It is not a LangGraph `interrupt()`.
5. The agent profile must be publishable through the governed agent-definition
   path as `source_yaml -> compiled_config`, not only file-backed legacy config.

## Acceptance Criteria

- [ ] The Affidea profile validates both as file-backed config and through the
      governed agent-definition compiler.
- [ ] Front desk routes FAQ, consultation booking, MRI booking, human handoff,
      and end-call intents through explicit graph routes.
- [ ] Consultation can complete a local happy-path booking using Affidea tool
      aliases and reused prompt text.
- [ ] MRI can complete a local happy-path booking and can route unsafe answers
      to handoff with an operator-safe summary.
- [ ] Auth can resume to the originating consultation or MRI flow after
      successful identity verification.
- [ ] Browser rehearsal starts the Affidea profile without changing the existing
      clinic-registration profile defaults.
- [ ] Phone routing fails closed if the Affidea agent definition is unpublished
      or the appointment-booking solution is disabled for the tenant.
