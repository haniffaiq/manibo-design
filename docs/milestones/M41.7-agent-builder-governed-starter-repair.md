# M41.7: Agent Builder Governed Starter Repair

## Goal

Repair the new agent-builder UI so it stays inside the governed
starter/runtime contract. This milestone restores legacy deep-link behavior,
removes the frontend-owned starter creation path, aligns the builder with the
runtime `channels.voice.*` schema, and repairs the broken web test surface
that still targeted deleted pre-builder modules.

## Design Decisions

1. `#961` remains the reviewer-facing PR for this milestone. Relevant builder
   fixes from `#962` are folded into it, while unrelated M40 call-ops work
   stays out.
2. Governed starters stay backend-owned. New assistant creation must use
   `/admin/agent-starters` filtered by tenant-enabled solutions, not static
   frontend templates.
3. The builder is a shell over authored YAML. When a starter is chosen, the
   first builder render patches that starter YAML instead of inventing a blank
   document, so governed keys survive the first save.
4. Legacy links stay valid. Existing `/admin/agent-definitions/:id` and
   `/test` links must preserve `tenant_id`, `definition_name`, `version`,
   `starter`, `source`, and the requested workbench tab.
5. Frontend-owned catalogs cannot invent production starter or tool IDs.
   Unsupported local catalogs stay hidden until a governed backend contract
   exists.

## Acceptance Criteria

- [ ] Creating a new assistant uses governed starters filtered by the selected
      tenant's enabled solutions.
- [ ] Legacy `/admin/agent-definitions/:id` and `/test` links preserve
      workbench state and land on the requested assistant, version, and tab.
- [ ] Builder save/load round-trips runtime STT/TTS settings from
      `channels.voice.stt` and `channels.voice.tts` without silently resetting
      them to defaults.
- [ ] Static frontend starter catalogs no longer advertise invalid production
      starter or tool IDs.
- [ ] The broken `apps/web/tests/**` builder coverage is replaced with passing
      tests that prove the new route, starter, and YAML behavior.
