## Goal

Surface the per-turn voice latency data that M8 already collects and persists into a live, operator-facing visualization inside the observability workspace. The backend already captures 29 fields per turn (STT finalize, LLM TTFT, TTS TTFB, end-of-turn to agent speak, interruption overlap, tool executions) and the API already returns them via `GET /calls/{call_id}/latency`. No frontend component renders this data today вҖ” the `turns` array is fetched and discarded. This milestone closes that gap.

## Current State (verified in repo)

- `VoiceTurnLatency` dataclass: 29 fields per turn including all timing milestones, interruption tracking, and tool executions (`packages/grove/src/grove/temporal/voice_call_models.py:67-96`)
- `VoiceTurnLatencyCollector`: state machine tracking user_speak вҶ’ stt_finalize вҶ’ llm_start вҶ’ llm_ttft вҶ’ tts_ttfb вҶ’ agent_speak per turn, plus interruption overlap (`packages/grove-voice-livekit/src/grove_voice_livekit/voice_latency_collector.py`)
- `GET /calls/{call_id}/latency` returns `CallLatencyResponse` with `turns: list[LiveCallTurnLatency]` + `summaries` + `stack` (`apps/api/src/platform_api/routes/calls.py:1671-1710`)
- Frontend `LiveCallTurnLatency` interface has 11 fields вҖ” subset of backend's 29, with 2 field name mismatches (`apps/web/src/lib/api/call-observability.ts:24-35`)
- `SupportLatencyMetrics` renders only 4 aggregate metric cards from `summaries` dict вҖ” never touches `turns` array
- `useLiveCaseStream` returns merged transcript + runtime events but no latency data
- `LiveCallTelemetrySignal` sends 5 most recent turns during live calls; `CallCompletedSignal` sends all finalized turns
- M1 decomposed observability into clean components: `EvidenceRail`, `CaseRecordPanel`, `AudioWaveformTimeline`, `CaseHeader` вҖ” all have clear insertion points

## Problem Statement

The platform collects detailed per-turn voice latency data but operators see only 4 aggregate numbers (average/worst/latest across the whole call). When a call degrades on turn 7 out of 12, the operator has no way to see that. When STT is fast but LLM is slow on one specific turn, it's invisible. The data exists in the database and API вҖ” the UI just doesn't render it.

## Design Decisions

1. **Unified turn rows, not separate waterfall + transcript** вҖ” the original design had a standalone waterfall section above the evidence timeline AND separate transcript items below. This is duplication: each waterfall bar maps 1:1 to a transcript item. The merged design puts the latency bar inline on each transcript turn row, eliminating redundancy.
2. **Inline expand, not sidebar detail card** вҖ” clicking a turn expands it in-place to show the full pipeline breakdown and timing milestones. This is faster than jumping eyes to a sidebar panel. The right sidebar keeps its existing role (case record, actions, trace context).
3. **Live + historical in one component** вҖ” during live calls, new turns appear at the bottom via SWR polling of `GET /calls/{call_id}/latency`. After call completion, it shows all turns with finalized data.
4. **Interruption and tool overlays** вҖ” turns where the caller interrupted the agent get a visual marker. Tool executions within a turn show as nested items below the latency bar.
5. **Extend frontend types, don't reinvent** вҖ” align `LiveCallTurnLatency` with backend fields. Fix the 2 field name mismatches. Add missing fields needed for visualization.
6. **Pure CSS/Tailwind bars** вҖ” no charting library. Inline styles for bar widths proportional to max `eot_to_agent_speak_ms`. Keeps bundle size zero-delta.
7. **Respect M1 component boundaries** вҖ” new components live in `apps/web/src/components/observability/`. The evidence timeline's existing item rendering is extended, not replaced.
8. **System events stay separate** вҖ” routes, nodes, metrics, logs are NOT per-turn. They stay in a "System Events" section below the conversation turns.

## Non-Goals

- No changes to backend latency collection or persistence (M8 already handles this)
- No new API endpoints (existing `GET /calls/{call_id}/latency` has all data needed)
- No charting library (D3, Chart.js, Recharts) вҖ” pure CSS/Tailwind bars
- No aggregate metric redesign (the existing 4-card summary stays; this adds per-turn detail)
- No changes to SupportDrawer or call-ops views (observability workspace only)
- No WebSocket transport for latency data (SWR polling is sufficient for 3s refresh during live calls)
- No sidebar detail card for selected turns (inline expand instead)

## M33 Impact

**Enables new scope.** Autonomous voice agents introduce new latency sources: memory lookup, PTC child process overhead, and context compression between turns. M8.1 visualization can expand to show these autonomous-specific sources alongside the existing STT/LLM/TTS pipeline breakdown. No changes required if focusing on rail agent path only.
