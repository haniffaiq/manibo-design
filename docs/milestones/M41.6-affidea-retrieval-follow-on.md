## Goal

Add smarter retrieval only if M41.4 eval evidence proves the deterministic KB
version is not enough. This milestone covers design-doc migration Phase 4 and
keeps vector or chunked retrieval out of the first booking-critical release by
default.

## Design Decisions

1. This milestone is parked until M41.4 produces concrete eval misses that
   deterministic KB files cannot reasonably fix.
2. Booking-critical IDs, slots, patient data, and provider writes still come
   from tools and graph state, not retrieval.
3. Retrieval, if activated, is limited first to FAQ or large unstructured
   reference material. It must not replace deterministic provider-derived KB
   artifacts for booking decisions.
4. Activation requires a short research note naming the failed scenarios,
   expected lift, latency risk, and rollback path.

## Acceptance Criteria

- [ ] Retrieval is activated only with M41.4 evidence, not as speculative
      infrastructure.
- [ ] Booking-critical reference IDs remain deterministic and test-covered.
- [ ] Retrieval output is separated from live data and protected patient state.
- [ ] Evals prove retrieval improves the documented failure cases without
      regressing booking correctness or latency.
- [ ] Operators can disable retrieval and fall back to deterministic KB files.

## Non-Goals

- No retrieval work before M41.4 evidence exists.
- No patient-data retrieval.
- No replacement for live slot/auth/booking tools.
- No LangGraph Platform server adoption.
