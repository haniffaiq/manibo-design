## Goal

Move E2E and regression tests off k3d-in-CI onto a persistent single-node k3s
staging cluster on Hetzner. Staging tracks `main` via Flux; every merge to main
auto-deploys to staging and runs the real E2E suite against a real cluster.
Promotion to production is gated on staging green and auto-merges via the
existing release-pin PR, unless a `blocks-auto-promote` label is set on the
source PR for dangerous changes. This closes the 50% k3d infra flake we
measured on both merge-gate and the release-deploy path, drops push-to-prod
wall-clock from ~3h to ~45m, and raises release success rate from 38% toward
90%+ without adding a branch model.

## Current Status Snapshot

1. Merge-gate `Run Full K8s Runtime Proof` runs k3d in CI on a self-hosted
   heavy runner (~35m/run, measured 2026-04-15 on run 24463062967). Half the
   time on the release path the same k3d spin fails before any test runs
   (kernel modules, k3d cluster never ready, port 5060 collisions).
2. `.github/workflows/flux-production-deploy.yml` runs a second `Run Full K3d
   E2E` (~31.6m) on the prod-release branch before flux push. That second spin
   failed 22 of 40 recent runs; 12 of those 22 were pure infra flake that
   merge-gate already proved green on the same SHA.
3. Push-to-prod wall-clock today: typically 2-3h, worst ~3.5h. Release success
   rate 38% over the last 40 attempts.
4. Prod Hetzner cluster has 12 nodes (3 CP + 9 workers across 5 pools) plus
   external monitoring + vault + deploy-runner hosts. Staging does not exist.
   Staging cost target: ~€10/month on the 2026-04-16 Hetzner pricing
   snapshot (1x cx33 ~€7 + Floating IPv4 ~€3; no LB, no separate Object
   Storage subscription, no external ops hosts). cx33 is the current
   orderable replacement for the deprecated cx32; same shape. Verify
   current Hetzner list prices before budgeting; they drift.
5. Prod uses SOPS-encrypted secrets with Vault Transit for key unsealing.
   Staging does not need Vault; SOPS with age keys is sufficient and keeps
   the staging VM count at one.
6. Manibo has two deployment targets: Hetzner (manibo) and GKE (NFQ). Staging
   lives on Hetzner because it validates platform code, not cluster-flavour
   behaviour. NFQ-specific validation stays in the existing NFQ merge path.

## Design Decisions

1. **Single-node k3s on one Hetzner cx33 box.** 4 vCPU / 8 GB RAM / 80 GB
   NVMe, ~€7/month. Co-locates k3s server, all platform workloads, CNPG,
   Temporal, LiveKit, agent-worker. Not HA by design; staging is disposable
   and rebuildable. cx33 is the current orderable replacement for the
   deprecated cx32 (same shape). Chosen over cx43 (~€14) because staging
   validates platform code, not performance under prod load. If 8 GB turns
   out tight (workload requests sum to ~10 GB; cx33 will rely on overcommit),
   bump to cx43 via a one-line terraform diff.
2. **One CNPG cluster with two databases.** Instead of separate
   `platform-postgres` and `temporal-postgres` CNPG clusters, staging runs a
   single 1-instance CNPG cluster that owns both `grove` and `temporal`
   databases. Halves the stateful surface, fits comfortably on the one node.
3. **SOPS + age keys, no Vault.** Staging keeps the SOPS secret pattern but
   drops the Vault Transit backend. Age keys live in a 1Password item and are
   loaded into the GitHub Actions runner via secret. No staging Vault VM.
4. **Prometheus runs in-cluster, no external monitoring VM.** Staging
   deploys Prometheus + Alertmanager as pods with short retention (3-5
   days), plus `remote_write` to the prod Prometheus so dashboards are
   durable across staging rebuilds. Saves ~€6/month on an external VM and
   keeps staging observability recoverable if the staging node dies.
5. **Flux watches `main` directly for staging.** Every push to main
   auto-reconciles staging. No ref pin. Prod continues to watch
   `prod-release/{sha}` refs. Same Flux tool, two different reconciliation
   targets.
6. **No backups, no HA, no secret rotation on staging.** Staging is a test
   environment; if it corrupts, rebuild from a make target that reseeds it
   from a fixture dump. Saves operational weight.
7. **E2E runs post-merge against staging, not pre-merge in k3d.** Trunk-based
   development stays; the merge decision is made on fast checks only. If
   staging E2E fails on a SHA, main is briefly red; we revert on main or
   roll forward. Acceptable because:
   - Merge-gate already catches product regressions in unit + integration
     + contract tests (no loss of pre-merge signal for 99% of PRs).
   - Per-PR preview environments can come later (Phase 2, separate milestone)
     if the "red main" window becomes operationally painful.
8. **Release-pin auto-merge, opt-out with a label.** Today the release-pin
   PR requires human merge. We flip the default to auto-merge gated on
   staging E2E green, release-pin required checks, and the absence of a
   `blocks-auto-promote` label on the source PR. Label is author-applied
   for migrations, auth changes, schema breaks, or anything needing manual
   staging verification.
9. **Retire the duplicate `Run Full K3d E2E` on the release path.** Once
   staging E2E gates the release-pin PR, `flux-production-deploy.yml`
   collapses to the `deploy` job only. Release-deploy wall-clock drops from
   ~78m to ~15m.
10. **Keep local k3d for dev.** `tools/scripts/infra/k3d-up.sh` stays for
    engineers working locally. Only CI-embedded k3d dies.
11. **Staging is single-tenant by SHA.** No per-PR preview environments in
    M26.9. Staging reflects the current main SHA; concurrent PR surface
    testing is explicitly out of scope for this milestone. Revisit in a
    follow-on if merge velocity makes that painful.
12. **NFQ staging is not in scope.** NFQ uses GKE, not Hetzner. Their
    staging/deployment story is tracked in M38/M39. M26.9 validates
    Manibo-on-Hetzner only.
13. **Shared Object Storage subscription with prod, different key prefix.**
    Staging terraform state lives in the same Hetzner Object Storage
    project as prod (`tf-state-manibo-production-nbg1`), under a
    `staging/` key prefix. Saves the ~€6/month minimum fee for a second
    Object Storage subscription. IAM is scoped via a separate access-key
    pair with path-prefixed policy, so a staging operator cannot
    overwrite prod state.
14. **No load balancer on staging.** Hetzner LB11 (~€6/month) is
    overkill for one node. Ingress-nginx runs on `hostNetwork` and binds
    to the single node's public IPv4 + IPv6 directly. A Hetzner Floating
    IPv4 (~€3/month on the 2026-04-16 pricing snapshot) stays attached
    to the node so DNS can point at a stable target even if the VM is
    rebuilt.
15. **Public firewall open for PSTN + RTC media.** Staging accepts real
    inbound phone calls via Telnyx so we can test LiveKit SIP + RTC with
    production-representative traffic. Same voice firewall rules as prod
    apply (SIP 5060 UDP/TCP, SIP-TLS 5061 + 5349, RTP 10000-20000 UDP,
    LiveKit RTC 50000-60000 UDP). HTTPS/HTTP on ports 80/443 via
    ingress-nginx on hostNetwork. SSH admin gated by `admin_cidrs`.

## Acceptance Criteria

- [ ] Staging cluster exists, Flux is reconciling main to it, and
  `platform-api` responds 200 on `/healthz` through the staging ingress.
- [ ] A merge to main triggers staging deploy + staging E2E within 20
  minutes and reports back as a required check on the release-pin PR.
- [ ] Release-pin PR with no `blocks-auto-promote` label and green staging
  E2E auto-merges without human touch. Flux-production-deploy fires on
  merge and deploys in under 15 minutes.
- [ ] Release-pin PR with `blocks-auto-promote` label holds until the
  label is removed; auto-merge resumes on label removal.
- [ ] Merge-gate `Run Full K8s Runtime Proof` is removed. PR-scope
  merge-gate wall-clock drops below 15 minutes for a typical frontend PR.
- [ ] `flux-production-deploy.yml` has no `Run Full K3d E2E` job. Its
  only remaining job is `deploy`.
- [ ] 20 consecutive merges to main: staging deploys green, release-pin
  auto-merges, prod deploys green. Release success rate ≥90% measured
  across this window.
- [ ] Push-to-prod wall-clock median drops from ~3h to ≤60 minutes measured
  across 10 consecutive merges.
- [ ] No same-SHA staging E2E reruns needed on 10 consecutive merges (flake
  guardrail).
- [ ] Staging rebuild from fixture via `make staging-reset` completes in
  under 5 minutes and leaves staging in a known-good seeded state.

## Non-Goals

- **No per-PR preview environments.** Staging reflects main only. Per-PR
  namespaces can be a separate follow-on milestone if merge velocity
  justifies it.
- **No staging HA.** Single node. If it dies, rebuild.
- **No staging backups, retention, or disaster recovery.** Staging is
  rebuildable from fixtures; no Barman, no S3 retention.
- **No staging Vault.** SOPS + age keys only.
- **No staging NFQ/GKE cluster.** M26.9 is Manibo-on-Hetzner only. NFQ's
  staging story is tracked independently.
- **No branch model changes.** Single trunk (`main`) stays. Promotion is
  ref-based via `prod-release/{sha}`, same as today. We do not add a
  long-lived `staging` branch.
- **No changes to merge-gate's scope-classification logic.** The `changes`
  job keeps doing what it does; we only remove the heavy k3d job after it.
- **No vCluster migration.** Rejected 2026-04-15
  (`wiki/queries/2026-04-15-ci-speed-vcluster-vs-parallelism.md`). Staying
  with real k3s on one VM.
- **No CNPG streaming replication on staging.** Single CNPG instance, two
  databases in one Postgres process.
- **No feature flag for the rollout.** Staging comes up, workloads deploy,
  E2E workflow gates the release-pin — the sequence is the feature gate.
  If something breaks, flip the `Run Full K3d E2E` job back on in
  flux-production-deploy and we're back to the current behaviour while we
  fix forward.

## Risk Register

| Risk | Mitigation |
|------|------------|
| Staging cluster is single-node; if the VM dies, staging is down and blocks all releases. | `blocks-auto-promote` lifecycle already supports "pause promotion"; use it during staging outage. `make staging-reset` rebuilds from fixture in <5m on a new VM. |
| Staging tracks main continuously; a bad merge makes staging red and blocks every subsequent release until fixed. | Short revert playbook: `git revert` the bad commit on main, flux reconciles staging, staging recovers. Expected <15m. |
| Staging E2E flake blocks release-pin auto-merge. | Flake guardrail in acceptance criteria (no same-SHA reruns in 10 merges). If flake appears, the fix lives with the test, not the harness. |
| SOPS age key leak from the staging runner. | Age key is read-only for staging secrets, scoped to one environment, rotatable in <1h. Prod Vault Transit is untouched. |
| NFQ/GKE release path diverges from Manibo/Hetzner release path. | Documented non-goal. NFQ has its own release pipeline (M38/M39); M26.9 does not attempt a unified release contract. |
| Cost creep: €30/month now, €X in 3 months as staging grows. | Single-node is the explicit "start minimal" choice. A move to 2-node is a terraform diff, not a rebuild. Monthly cost is tracked in the staging README next to the node pool spec. |
