## Goal

Turn the M4 public chat widget from a minimal product slice into a managed web-chat runtime. Add operator-visible runtime state, guest-session ownership, routing/thread truth, and explicit runtime health/status read surfaces so web chat behaves like an operable channel instead of random widget glue. Reuse the existing `/observability/channel-runtimes*` and `/admin/observability/channel-runtimes*` read surfaces unless later inventory work proves a real gap. Recovery commands remain a follow-on only after the control-plane command contract is real.

- Phase 1 website sales work already requires the public chat widget, secure anonymous ingress, and conversational lead-qualification durability.

- This doc is kept in the milestone catalog only to reserve naming, boundaries, and dependencies for later runtime work; it is not activation-ready until an explicit checklist row and tracker decision make it real.

## Design Decisions

1. **M4 stays the minimal product slice** -- M14.2 operationalizes the runtime after the widget/bootstrap/session basics exist.
2. **Guest sessions become explicit runtime-owned state** -- session/thread truth lives in the channel runtime model, not only in widget localStorage or ad hoc request flow.
3. **Runtime status is operator-visible** -- health, backoff, auth/configuration state, and delivery problems must be inspectable without code spelunking.
4. **Web chat runtime reuses the shared channel primitives from M14.1** -- do not invent a second session/state model just for the widget.
5. **Read surfaces bind to existing observability routes first** -- this planning lane targets the existing `/observability/channel-runtimes*` and `/admin/observability/channel-runtimes*` route families; inventing a new API family is out of scope unless regenerated inventory proves a real missing consumer.
