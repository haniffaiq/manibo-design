# M26.11 — Staging cluster bootstrap

## Context

M26.9 landed the staging Flux scaffolding, kustomize overlay, CNPG +
Temporal rewire, in-cluster Prometheus, staging-e2e workflow, release-
pin gate wiring, and the staging-reset Make target. Multiple review
rounds surfaced that the overlay-only landing cannot by itself make
staging reconcile cleanly or emit a trustworthy `staging-e2e` signal:

- `platform-runtime-secrets` and `platform-web-runtime-secrets` are
  referenced by base workloads but not seeded on staging. Pods
  CrashLoop/Pending without them and Flux times out with `wait: true`.
- No SOPS age key exists on the staging cluster, so `sops-age` Secret
  references (`decryption.provider: sops`) cannot resolve. M26.9
  intentionally left that stanza out; re-introducing it is a scope of
  M26.11.
- Staging reconciles `flux/clusters/production/apps/release-images.yaml`
  from main, not the release-pin PR's image SHAs. The `staging-e2e`
  gate therefore tests the previous release-pin, not the candidate.
  Either staging needs its own image-pin file that the
  `publish-platform-images` workflow updates, or the staging-e2e
  workflow needs to post its status on the release-pin PR head SHA.
- The `staging-e2e` workflow's `push: main` trigger is disabled
  (M26.9 T02d) until M26.11 ships; the release-pin auto-merge gate
  is defunct in the meantime.
- Temporal DB credential lives as plaintext `stringData` in a
  Secret manifest (M26.9 T02c pragmatic landing); should be SOPS-
  encrypted against the staging age key.
- Alertmanager routes `env=staging` alerts to a `null` receiver;
  `StagingNodeNotReady` is silent.

M26.11 closes all of these so the staging cluster can be brought
online as a real gate without more deferred-work pointers.

## Goal

After M26.11 lands and the operator runs `flux bootstrap github
--path=infrastructure/kubernetes/flux/clusters/staging --branch=main`
against the staging cluster, the following must be true:

1. All Flux Kustomizations reconcile `Ready=True` within 10 minutes.
2. `platform-api`, `platform-web`, `temporal-frontend/history/
   matching/worker`, `agent-worker`, `temporal-worker`, MinIO, CNPG,
   ingress-nginx all report Ready.
3. `staging-e2e` workflow fires automatically on push to main and
   publishes a `staging-e2e` commit status on the reconciled SHA.
   Release-pin PRs are opened with auto-merge off and require manual
   merge after confirming a green `staging-e2e` on a recent main SHA
   (predecessor-main model; gating on the actual candidate SHA is a
   follow-up).
4. `make staging-reset` works end-to-end against the live schema.
5. `StagingNodeNotReady` alert forwards to a non-paging operator
   Slack channel (or explicit documented receiver), not `null`.
6. Every live-cluster credential in `infrastructure/kubernetes/
   overlays/hetzner/staging/` is SOPS-encrypted (zero plaintext
   stringData in committed Secret manifests).

## Design Decisions

1. **SOPS age key, not Vault Transit.** M26.9 DD #3 already decided
   this. M26.11 delivers it: `age-keygen` produces a keypair, private
   half installed on the cluster as the `sops-age` Secret in
   `flux-system` (that is the sole read path Flux needs), with the
   operator's local file (`~/.config/sops/age/manibo-staging.txt`,
   mode 600) as the long-lived source-of-truth backup. Public half
   committed to `.sops.yaml` against the staging path prefix. The
   `SOPS_AGE_KEY` GitHub env secret is intentionally NOT provisioned
   — no workflow reads it today, and storing an unprotected env-level
   private key is blast-radius without payoff. When a future workflow
   needs to decrypt staging SOPS files, add it then, with protection
   rules enforced (required reviewers + deployment-branch policy).
2. **Staging-specific `release-images.yaml`** at
   `infrastructure/kubernetes/flux/clusters/staging/apps/release-images.yaml`.
   `publish-platform-images.yml` updates both prod's and staging's
   pin files in the same release-pin commit so staging eventually
   reconciles the same SHA that prod will promote. The **original
   M26.11 goal** of posting `staging-e2e` on the release-pin PR head
   and gating prod auto-merge on it is explicitly deferred (T04
   shipped the predecessor-main model instead; see T04 Implementation
   Notes). Release-pin PRs are now opened with auto-merge OFF and
   require manual merge after the operator confirms a recent green
   `staging-e2e` on `main`. True candidate-head validation is a
   follow-up that requires Flux on staging to track a release-
   candidate branch instead of `main`.
3. **CNPG-managed temporal role with password from SOPS Secret.**
   postInitSQL creates the role without a password (`CREATE ROLE
   temporal WITH LOGIN`), and CNPG's `managed.roles[]` stanza syncs
   the password from `temporal-postgres-app` (SOPS-encrypted). The
   Temporal helm chart resolves creds via `existingSecret:
   temporal-postgres-app`, so the SOPS Secret is the single owner of
   the credential and there is no plaintext password literal in any
   committed manifest.
4. **Alertmanager staging receiver = Slack webhook.** Staging alerts
   are operator-hours, not pager-worthy. A dedicated low-noise Slack
   channel receives `env=staging` routes. `null` receiver retained
   as the catch-all fallback.
5. **Schema-aligned fixture.** `tools/scripts/staging/fixtures/
   staging-seed.sql` rewritten against the actual User + Tenant +
   Membership + TenantSolution + solution-owned agent + phone-number
   tables. Reset script's `STAGING_SEED_CONFIRM` guard is removed
   once the fixture is real. E2E tests can reference fixed UUIDs
   from the seed.
