## Goal

Move NFQ production runtime secret ownership from live-only Kubernetes Secrets
to GCP Secret Manager synced by External Secrets Operator, with automatic pod
rollouts when synced secret versions change.

## Design Decisions

1. **GCP Secret Manager is production source of truth.** SOPS remains useful for
   local/Hetzner overlays, but NFQ GCP production should use the provider-native
   secret store.
2. **ESO uses Workload Identity.** No JSON service-account keys are mounted into
   the cluster.
3. **Secret bundles are JSON objects.** ESO extracts one JSON bundle into one
   Kubernetes Secret. This keeps the operational secret boundary aligned with
   existing application Secret names.
4. **Reloader handles pod restarts.** Deployments explicitly declare which
   Secrets and ConfigMaps trigger restart.
5. **Secret-sync controllers are monitored before launch.** ESO and Reloader
   are now part of the runtime secret-delivery path, so NFQ/GCP observability
   owns controller restart, controller availability, ExternalSecret readiness,
   and ClusterSecretStore readiness alerts.
6. **Legacy Kubernetes writes are blocked in GCP production.**
   `k8s-runtime-secrets.sh apply --overlay gcp/production` refuses to write the
   ESO-owned runtime Secrets; operators must seed or rotate the Secret Manager
   bundles instead.
