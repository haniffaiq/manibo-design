## Goal

Reserve the WhatsApp interactive-runtime boundary for future work without smuggling it into the current requirement contract. The current explicit checklist row for preferred-channel WhatsApp notifications stays owned by `M14` request/response transport, not by this milestone. This milestone only exists for a future full bidirectional WhatsApp product surface and stays planning-only until an explicit interactive-WhatsApp checklist row and endpoint contract exist, plus the active tracker explicitly activates it.

## Requirement Anchors

- VOX `REQ-P03` requires preferred-channel notifications after schedule changes.
- That row alone does not justify inbound WhatsApp message handling, session/thread ownership, provider account install/auth flows, or runtime UI.
- Interactive WhatsApp stays out of executable scope until the requirement contract grows beyond outbound preferred-channel delivery.

## Design Decisions

1. **WhatsApp is an interactive channel runtime, not a connector** -- account auth, webhook lifecycle, runtime state, and delivery status live in the channel-runtime lane.
2. **WhatsApp transport lives in Layer 3** -- shared runtime contracts stay in a future Layer 2 channel-runtime package family under `packages/platform-core/` (exact path TBD until activation), while WhatsApp-specific transport/auth/webhook behavior lives in an installable channel package/provider pack.
3. **Notification routing is policy-driven** -- email/SMS/WhatsApp routing is selected by tenant policy rather than hardcoded per workflow.
4. **WhatsApp builds on the already-proven channel model** -- do not start until M14.1 and M14.2 make the generic runtime contract honest.
5. **Unified routing depends on M14 transports** -- email/SMS policy routing is dishonest unless M14 lands the shared notification adapters first.
6. **Reuse the existing generic runtime detail/timeline routes first if this ever activates** -- tenant/admin detail and timeline routes already exist at `/observability/runs/channel_runtime/{runtime_id}` and `/admin/observability/runs/channel_runtime/{runtime_id}`. Future activation must start from those generic runtime surfaces instead of inventing duplicate detail APIs.
7. **Do not treat widget routes as generic runtime contracts** -- `/observability/channel-runtimes*` and `/admin/observability/channel-runtimes*` are widget-scoped read surfaces reserved for M14.2 web-chat runtime operations. Future WhatsApp work must not overload `widget_id` or force WhatsApp into widget-shaped APIs.
8. **Only add WhatsApp-specific read contracts if the current generic surfaces prove insufficient** -- any future gap must be limited to genuinely missing non-widget filters, fields, or consumer contracts keyed by an honest runtime identifier, and must be added to the inventory before task files are created.
