## Goal

Create the first real NFQ GCP staging environment and use it to prove
self-hosted LiveKit with real PSTN calls. Staging must be separate from
production, use GCP Secret Manager as the runtime-secret source of truth, and
produce latency evidence that can be compared against the existing production
LiveKit Cloud baseline.

## Design Decisions

1. **This is NFQ/GCP staging, not Hetzner staging.** M26.9 remains the Manibo
   Hetzner staging track. M38.3 owns the NFQ GKE staging target.
2. **Staging uses self-hosted LiveKit.** Production starts on LiveKit Cloud.
   Staging gives us the comparison target before deciding whether production
   should move.
3. **Secret Manager is the staging source of truth.** Do not introduce SOPS as
   the GCP staging runtime-secret source unless this decision is reopened.
4. **No production resource reuse.** Staging needs dedicated DNS, Secret
   Manager bundles, Telnyx connection/trunks/DID, LiveKit resources, and tenant
   runtime seed data.
5. **Everything durable is IaC/GitOps-friendly.** Terraform owns GCP
   resources. Kubernetes manifests own runtime overlays. Manual discovery steps
   must be codified before the milestone closes.
6. **NFQ/GCP tests stay namespaced.** New architecture tests for this work live
   under `tests/architecture/nfq/gcp/`.

## Acceptance Criteria

- [ ] Staging uses a dedicated NFQ GCP project or an explicitly documented
      staging-isolated project boundary approved before apply.
- [ ] All five staging Terraform roots validate:
      `project_bootstrap`, `network`, `platform`, `workloads_bootstrap`, and
      `google_runtime`.
- [ ] `infrastructure/kubernetes/overlays/gcp/staging` renders with no
      production hostnames, production Secret Manager names, or production DIDs.
- [ ] Staging runtime Secrets are sourced from GCP Secret Manager through ESO,
      and Reloader annotations roll affected pods.
- [ ] Self-hosted LiveKit Server and LiveKit SIP run in staging and expose the
      required SIP/RTP/WebRTC reachability.
- [ ] Staging uses a dedicated Telnyx DID and LiveKit SIP resources.
- [ ] One live inbound call and one live outbound call succeed through staging.
- [ ] Latency evidence is captured in the same style as the production LiveKit
      Cloud debug log so the two modes can be compared.

Live verification after deployment must record:

- GKE cluster and workload readiness
- ESO, Reloader, and ExternalSecret readiness
- self-hosted LiveKit and LiveKit SIP readiness
- Telnyx route and dedicated DID ownership
- inbound call proof
- outbound call proof
- latency comparison against the production LiveKit Cloud baseline

## Non-Goals

- Do not switch NFQ production to self-hosted LiveKit in this milestone.
- Do not reuse production runtime secrets, phone numbers, or LiveKit Cloud
  resources.
- Do not build the full M13.1 autonomous telephony evaluator here.
- Do not claim customer launch readiness from staging proof alone.
