## Goal

Move `solutions/appointment_booking` into the target business-capability
package structure before Affidea prompt, tool, and runtime work lands. This is
a behavior-preserving refactor: it makes room for Affidea without mixing new
booking logic with the existing clinic-registration starter.

## Design Decisions

1. This milestone owns the file-structure refactor from design-doc section
   8.5. Later M41-series milestones add Affidea behavior into this structure.
2. The refactor is structure-first and behavior-preserving. Existing
   clinic-registration config, plugin registration, tests, and API behavior
   must stay green.
3. Business capability code moves under solution-owned packages:
   `booking/`, `handoff/`, `automation/`, `crm/`, `voice/`,
   `observability/`, `evaluation/`, and `data/`.
4. The FastAPI route surface may move to
   `apps/api/src/platform_api/routes/appointment_booking` only as a thin shell.
   Domain behavior stays in `solutions/appointment_booking`.
5. Temporary compatibility shims are allowed only when import churn would make
   the PR harder to review. Each shim must be tiny, covered by tests, and have
   an explicit removal target in a later M41 task.
6. No Affidea-specific provider calls, grouped-flow runtime work, or prompt
   import work belongs in this milestone.
