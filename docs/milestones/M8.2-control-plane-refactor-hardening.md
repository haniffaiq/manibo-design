## Goal

Turn the existing call/control-plane and voice-runtime code from "it works, but the seams hurt" into maintainable module boundaries while the same seams are still small enough to cut cleanly. The current repo already contains duplicated workflow state, route junk drawers, split manual-takeover ownership, a monolithic LiveKit entrypoint, and UI/runtime/provider drift. M8.2 is the refactor-only follow-on that captures those hot spots without changing the user-facing product surface.

## Problem Statement

Even without M8 activation, the implementation tax is already visible in the current repo:

1. Review comments keep landing on boundary mistakes, not feature mistakes.
2. File-size gates are now hitting the exact modules where responsibilities are mixed.
3. The same operator-call semantics are implemented in too many places.
4. Adding new LiveKit providers or new operator UI surfaces will keep reopening the same seams unless ownership is simplified first.

This milestone is therefore not "cleanup for aesthetics". It is the smallest refactor slice that reduces change tax on the existing M8 surfaces and the planned turn-latency observability follow-on.

## Design Decisions

1. Extract logic before splitting files. Splitting one junk drawer into five smaller junk drawers is trash.
2. Prefer plain functions, dataclasses, and protocols over inheritance. Shared workflow behavior must stay deterministic and boring.
3. Route files stay thin. Projection, query, and orchestration logic must move out of `apps/api/src/platform_api/routes/**`.
4. One lifecycle owner per operator command. HTTP endpoints authorize and normalize input; a coordinator owns dispatch/audit truth; workflows emit generic terminal truth that Layer 2 adapts into control-plane semantics.
5. One authority for voice capability metadata. Runtime mapping and editor option lists must stop drifting independently.
6. Web consumers depend on a control-plane client contract, not on bridge URL strings and ad-hoc SSE fallback code.
7. M8.2 does not add new product behavior. If a task smells like a new feature, it is out of scope.
