## Goal

Build a first-class telephony evaluation framework that can place real inbound and outbound PSTN calls without a human handset, drive deterministic conversation scenarios, capture end-to-end evidence, and score conversation quality automatically for both `LIVEKIT_DEPLOYMENT_MODE=self_hosted` and `LIVEKIT_DEPLOYMENT_MODE=cloud`.

This is not a script pile. It is a product-owned runtime and test subsystem with persisted definitions, persisted runs, Temporal orchestration, provider adapters, and reviewable evidence.

## Why this exists

The current telephony control plane can now answer real calls, but the remaining failures are conversation-quality failures rather than routing failures:

- calls answer and media flows, but conversation quality regressed into slow pauses, cut words, and empty turns
- human handset testing is slow, inconsistent, and hard to compare across changes
- current observability now exposes turn-level metrics, but there is no autonomous caller/evaluator loop that can reproduce and score regressions without another human dialing
- the system needs a deterministic proof surface that can tell the difference between:
  - transport failure
  - webhook/control-plane failure
  - speech pipeline latency
  - interruption churn
  - TTS truncation
  - semantic task failure

## Current lessons that must not be lost

1. **Transport was not the final blocker**
   The self-hosted local SIP path was broken by a local `rtc.force_tcp: true` override in the LiveKit Server config. Removing that restored real PSTN answerability on the laptop.

2. **Cloud mode had a real provisioning bug**
   Telnyx inbound still routed to the old laptop edge even after a LiveKit Cloud SIP FQDN existed. The fix belonged in `tools/scripts/infra/setup_livekit_sip.py`: create the Cloud FQDN record and repoint the live Telnyx connection defaults.

3. **The remaining problem is now conversation quality**
   Calls answer, but the live conversation loop becomes slow or fragmented. The strongest runtime signatures from real calls were:
   - `stt end of speech received while user is speaking, resetting vad`
   - fragmented assistant outputs
   - empty follow-up turns
   - an `aiohttp ClientSession.close()` cross-event-loop cleanup error during call teardown

4. **Bootstrap observability was not enough**
   TTFT and room-start signals helped prove call setup, but they were not enough to explain conversation-quality failures. The system needed whole-turn metrics, interruption churn metrics, and turn-quality logs/spans before further debugging made sense.

## Design decisions

1. **This is an application subsystem, not a `tools/scripts` feature**
   Core logic lives in `packages/platform-core`, orchestration in `apps/temporal-worker`, and API surfaces in `apps/api`. Thin wrappers are allowed later, but `tools/scripts/**` cannot be the source of truth.

2. **Deterministic synthetic caller comes before AI judging**
   V1 uses scripted prompts, timing, thresholds, and expected outcomes. AI evaluation can be layered on top later, but deterministic scenarios are the baseline needed for reproducible regressions.

3. **Telnyx is the first provider adapter**
   Telnyx already exposes the primitives needed for autonomous PSTN evaluation: outbound call control, webhooks, media streaming, recording, and transcription. The shared protocol belongs in `platform-core`, but the Telnyx implementation belongs in the existing `solutions/provider_telnyx` layer so provider ownership stays in the solution pack that already owns Telnyx telephony behavior.

4. **Inbound and outbound are both required**
   Outbound evaluation must reuse the product-owned `POST /calls/test-call` path. Inbound evaluation must synthesize a PSTN caller dialing the assistant DID. Both directions must flow through the same result and scoring model.

5. **Persisted runs are the source of truth**
   Every evaluation run must persist:
   - definition id
   - run id
   - provider call identifiers
   - platform `call_id`
   - LiveKit room
   - scorecard
   - artifact metadata
   Results cannot live only in logs.

6. **Observability is part of the product contract**
   Evaluation runs must correlate with the existing metrics/logs/spans rather than invent a second observability stack. The scoring pipeline consumes the same `voice_application_turns_total`, `voice_turn_*`, and `voice_interruption_events_total` signals that human debugging already depends on.

7. **File-size discipline applies from the start**
   New files stay under 500 LOC by decomposition:
   - API route package split: `router.py`, `schemas.py`, `runtime.py`
   - evaluation domain split: `models.py`, `service.py`, `store.py`, `scoring.py`, `artifacts.py`
   - provider adapter split: shared `provider_protocol.py` in `platform-core`, Telnyx runtime files in `solutions/provider_telnyx`

8. **Live PSTN remains a scheduled/runtime proof, not a PR merge gate**
   The framework must support periodic canaries and on-demand runs, but real PSTN cannot become a required PR gate in v1. The immediate goal is autonomous reproducibility and diagnosis, not CI theater.

## Acceptance criteria

- [ ] One deterministic outbound PSTN evaluation can be launched without a human handset
- [ ] One deterministic inbound PSTN evaluation can be launched without a human handset
- [ ] Every run persists a correlated record linking provider call ids, platform `call_id`, and LiveKit room
- [ ] Every run produces a machine-readable scorecard with transport, latency, interruption, and semantic outcome sections
- [ ] Every run stores enough evidence to explain a failure without immediately rerunning the call
- [ ] The framework works against both `self_hosted` and `cloud` LiveKit deployment modes
- [ ] New runtime files created by this milestone stay under 500 LOC each
- [ ] The implementation reuses the existing voice observability surfaces rather than creating a second metrics stack

## Non-goals

- building a broad UI before the runtime and scoring model exist
- making real PSTN a required merge-gate CI lane
- pretending a generic multi-carrier abstraction exists before it is earned
- replacing the operator console
- hiding failures behind one opaque pass/fail number
