## Goal

Replace the current review harness with the simplest thing that works: one namespaced review entrypoint under `tools/scripts/review/`, one visible review verdict, and Codex-managed internal personas via project-scoped `.codex/` config and custom agents. Python should only handle deterministic plumbing: collect PR/MR context, pass a compact case file to Codex, parse one final result, and publish one authoritative review. Codex should handle the parallel specialist review work.

## Current Status Snapshot

1. Same-head reruns are worth keeping. One pass does not reliably find every issue, so the system must preserve unresolved findings and let the current head be reviewed again without creating a second blocker.
2. The current harness is too spread out. Prompt generation, GitHub fetching, review execution, parsing, and publishing are split across `tools/scripts/build_pr_review_prompt.py`, `tools/agents/review.py`, and `tools/agents/pr_review_bot.py`.
3. The first branch slice already extracted some durable pieces under `tools/agents/reviewbot/`: finding parsing and case-file reconciliation. Those helpers may stay only if they still earn their keep after the script-based rewrite. Everything else should move to `tools/scripts/review/` or be deleted.
4. The previous milestone draft over-designed the system around a Python review framework. The reset target is smaller: namespaced scripts plus `.codex/` config and agents.
5. Hooks are not the core design. They may become guardrails later, but the review bot itself should not depend on experimental hook behavior.

## Design Decisions

1. **One public review authority only.** The platform publishes one merge-blocking verdict per head. Internal personas do not create extra public bots or competing review states.
2. **Codex handles persona orchestration.** The parent review run asks Codex to spawn specialist subagents in parallel, wait for them, merge findings, and return one consolidated result.
3. **Python stays deterministic and boring.** The script layer collects repo/PR context, passes the case file to Codex, parses the final output, and publishes one review. It does not reimplement agent orchestration.
4. **Scripts stay namespaced under `tools/scripts/review/`.** No new flat scripts. Any new review plumbing lives under that directory.
5. **Repo-scoped Codex behavior lives in `.codex/`.** Project config goes in `.codex/config.toml`. Review personas go in `.codex/agents/*.toml`.
6. **The durable case file stays explicit.** Continuity comes from prior unresolved bot findings, not hidden model memory. Same-head reruns update the same authority. New-head reruns still read unresolved history.
7. **Project context is retrieved, not preloaded.** The parent prompt instructs Codex to inspect the diff, read tests first, read repo docs and instructions, inspect unresolved findings, and then retrieve additional code as needed.
8. **Deterministic evidence stays deterministic.** Dependency, license, and CVE signals may feed the final review, but they are not a fourth free-form reviewer persona.
9. **GitHub first, GitLab second.** The first cut keeps GitHub working and simple. GitLab support is added through a small adapter, not by forking the review logic.
10. **No new review framework unless forced by reality.** If `run_pr_review.py` plus small adapters can hold the behavior, do not build `engine.py`, `runner.py`, or similar abstraction layers.
11. **Fail closed.** Ambiguous final output, parser failures, or publish failures are not clean reviews.
12. **Delete glue aggressively.** Prompt builders, duplicate wrappers, and tiny helpers that only move strings around should disappear as soon as the simple path is proven.

## Target System Shape

```text
+----------------------------+
| PR/MR head update          |
+----------------------------+
              |
              v
+--------------------------------------+
| tools/scripts/review/run_pr_review.py|
| - collect diff and prior findings    |
| - load parent review prompt          |
| - call Codex once                    |
| - parse one final result             |
+--------------------------------------+
      |                         |
      v                         v
+--------------------+   +----------------------+
| github.py / gitlab |   | .codex/config.toml   |
| fetch + publish    |   | .codex/agents/*.toml |
+--------------------+   +----------------------+
                                   |
                                   v
                     +------------------------------+
                     | Codex parent review run      |
                     | spawns reviewer subagents    |
                     | merges findings              |
                     +------------------------------+
```

## Acceptance Criteria

- [x] Review runs stay stateless, but same-head and new-head reruns still evaluate unresolved prior findings through an explicit case file.
- [x] The required review path publishes exactly one authoritative verdict per head, even when Codex spawns multiple internal reviewer agents.
- [x] New deterministic review plumbing lives under `tools/scripts/review/` rather than in new flat scripts or a large Python framework.
- [x] Project-scoped Codex behavior lives in `.codex/config.toml` and `.codex/agents/*.toml`.
- [x] The parent review prompt tells Codex to spawn the generalist, security, and architecture reviewers, wait for them, and return one merged result.
- [x] GitHub support preserves current-head review markers, durable summary behavior, and same-head rerun semantics.
- [x] NFQ can run the same review flow with an API-key-backed Codex profile without forking review logic.
- [x] GitLab support can fetch MR context and publish one authoritative merged review result without inventing a second review authority.
- [x] Deterministic dependency/license/CVE evidence can feed the final review result without becoming another free-form reviewer.
- [x] The final code path is visibly smaller and easier to reason about than the starting harness.

## Final Notes

1. The shared local and GitLab workflow entrypoints now live under `tools/scripts/review/`, while GitHub workflows still call `tools/agents/pr_review_bot.py` directly until the extraction is complete.
2. `tools/agents/pr_review_bot.py` remains the GitHub review implementation and workflow entrypoint because the repo's size/test guardrails are already anchored there.
3. `tools/scripts/build_pr_review_prompt.py` still exists because that implementation uses it internally.

## Non-Goals

- No second public review bot.
- No hook-driven review engine.
- No new `tools/agents/reviewbot/engine.py`-style framework unless the simple script path proves insufficient.
- No giant prompt builder that fetches GitHub context just to assemble markdown.
- No free-form LLM dependency reviewer replacing deterministic scanners.
