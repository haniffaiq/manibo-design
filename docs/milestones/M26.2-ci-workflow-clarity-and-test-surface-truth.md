## Goal

Make the CI surface understandable and truthful: workflow/job names must explain what runs and why, runner docs must distinguish labels/pools from physical hosts, browser and platform E2E coverage must map to real GitHub jobs instead of tribal knowledge, release-environment smoke must be explicit instead of implied by unrelated runtime lanes, and prerelease release control must use a real full `k3d` E2E gate instead of bootstrap theater.

## Design Decisions

1. **M26.2 does not reopen the single-live-PR-workflow decision from M26.** Pull requests still use one authoritative PR workflow. Naming and topology cleanup may extract or rename non-PR paths, but they must not reintroduce multiple live PR workflows.
2. **Workflow and job names must explain trigger + purpose.** `gate`, `harness`, and `admission summary` are opaque garbage; the Actions tab should tell a junior engineer what is running without opening YAML.
3. **Runner truth must separate labels from machines.** CI docs may describe runner labels/pools (`automation`, `bots`, `merge-heavy`, etc.), but they must not imply one physical machine per label when the real host inventory is smaller.
4. **E2E truth beats file inventory theater.** Every E2E suite must be either wired to a real GitHub job, explicitly manual-only, or retired from the claimed CI surface.
5. **`k3d` proves repo/runtime truth before merge and on mainline.** Release-environment smoke validates deployed staging/production behavior after rollout; it does not replace cluster-backed runtime proof.
6. **Fail fast, then spend money.** Cheap scope/static checks must run before long `k3d`, browser, or provider-backed lanes.
7. **Route by touched capability, not by fear.** PRs should run only the suites materially required by the changed surface instead of brute-forcing the whole repo.
8. **M26.2 is clarity + control-plane truth, not another CI feature land grab.** No autoscaler revival, no second review authority, and no fake “improvement” that just adds more workflows without deleting ambiguity.
9. **Production release control must pay for real cluster-backed proof.** A prerelease gate that only boots `k3d` and curls ingress is fake confidence. Before production deploy, release automation must run the full deterministic `k3d` E2E surface that the repo already treats as deployable truth.

## Current Workflow Truth

1. The repo does not suffer from a pure “too many jobs” problem; it suffers from mixed responsibilities that now at least have operator-readable names.
2. `PR + Mainline Proof` still carries PR fast validation, PR heavy proof, and mainline deployable proof in one file by design, but the job surfaces now make that split readable.
3. `Ops - Production Maintenance` is not CI validation at all. It is a manual production-maintenance workflow that runs Terraform plan/apply, Flux reconcile, recovery drills, legacy-backup retirement, and sealed recovery-bundle export.
4. Browser E2E truth is explicit and intentionally local-only today: `apps/web/e2e/*.spec.ts` is owned by `tools/scripts/e2e/run-web-e2e.sh` and `tools/scripts/review/pre-pr-ci.sh`, not by GitHub Actions.
5. Runner language is explicit: the repo uses several labels/pools, but that does not mean one physical machine per label.
6. Voice-room coverage truth is now explicit: the nightly/mainline flows invoke `packages/grove-voice-livekit/tests/e2e/test_voice_room_e2e.py` from the package working directory, and the docs no longer pretend that path is broken.
7. Active PR/mainline fast proof is GitHub-hosted, but the core VM still carries a self-hosted `control` runner service for manual replay and runner-health checks. Those are different truths and the docs must say so explicitly.

## Reduction Plan

| Surface | Decision | Why |
|---------|----------|-----|
| `Merge gate` | rename PR path; evaluate extracting non-PR path only | Keep one live PR workflow while separating non-PR intent if that reduces confusion without reviving PR workflow sprawl. |
| `Periodic Tests` | split | The deterministic name is currently lying because `periodic-gcp-smoke` uses real providers. Keep deterministic periodic separate from provider smoke. |
| `CI Duration Health` + `CI Scheduler Health` | rename separately, revisit merge later only if duplication becomes real | T04 proved naming/vocabulary cleanup was enough; forcing a merge here would have added churn without improving the actual fleet truth. |
| `export-check.yml` | keep short-term, consider merge later | It has real path-scoped value today; merge it only after path classification can route export proof without bloating the main workflows. |
| `PR admission` + `PR heavy` | delete after tooling migration | They are manual replay/debug surfaces only. Keeping them forever is fossilized CI clutter. |
| prerelease `k3d` proof vs release smoke | keep separate | prerelease `k3d` proves repo/runtime truth before deploy; release smoke proves the deployed environment after rollout. They solve different failure classes and both earn their keep. |
| Orphaned E2E suites | classify or retire explicitly | T05 locks every remaining non-GitHub suite to a manual-only or local-only owner instead of pretending CI runs it. |
| Browser E2E | keep local-only until a real GitHub lane exists | The repo can keep browser proof, but only if docs say the local harness owns it today. |
