## Goal

Simplify merge-critical CI by removing PR-scoped runtime proof altogether and making one mainline `k3d` run the authoritative cluster-backed runtime truth. The target shape is one `k3d` bootstrap per mainline proof run that executes the broad deterministic backend test surface, the traceability harness, and the checklist-derived `k3d` subset in the same run, while PRs keep only cheap static/product validation instead of paying repeated heavy-runner tax. “Broad deterministic backend” here means all backend tests we can honestly run in CI across `packages/*`, `apps/api`, platform-owned `apps/temporal-worker` surfaces, and deterministic `solutions/*` suites without live provider dependence, live LLM calls, or voice/telephony runtime dependence. Mislocated solution-owned Temporal worker E2E files stay out of this generic sweep and remain manual-only until they move under `solutions/*` or gain an explicit shared-ownership decision.

## Current Status Snapshot

1. PR runtime fan-out is removed. The live PR path keeps deterministic runner-shell validation plus optional artifact exclusion only.
2. `Run Full K8s Runtime Proof` now owns the traceability harness, broad deterministic backend unit/integration suites, and the checklist-derived `k3d` subset inside the same cluster-backed helper.
3. Release stays aligned by reusing the same full-runtime helper, and nightly remains the deep-regression owner without contradicting the new mainline truth.
4. Frontend proof is now GitHub-owned on both PR and mainline when frontend or coupled backend/API/auth scope is present.

## Design Decisions

1. **Runtime proof moves off PRs and onto mainline/release/nightly.** PRs keep fast deterministic non-runtime validation only. They do not run `k3d`, and they do not keep a separate compose runtime-smoke lane that pretends to be meaningful proof.
2. **One `k3d` bootstrap per mainline proof run.** This milestone does not create one persistent shared cluster. It reuses one cluster only inside the same workflow run and job; there is no cross-run/event/SHA cluster reuse fiction.
3. **Mainline full runtime proof owns traceability too.** If `Run Traceability Smoke` leaves the PR path, `merge-runtime-full` must absorb the traceability harness in the same implementation PR or coverage regresses.
4. **Checklist-derived `k3d` proof stays real, but loses its dedicated PR lane.** The checklist-driven subset remains part of authoritative cluster proof; it just stops living behind a separate PR job.
5. **Release stays aligned with mainline helper truth.** `Release - Deploy Production / Run Full K3d E2E` should keep sharing the same helper contract as mainline instead of drifting into a second definition of “full runtime proof”.
6. **PR fast lanes stay fast and non-runtime.** Repo/docs/static/frontend/admission surfaces stay on PRs, along with deterministic non-runtime backend suites that can run directly on the runner; runtime proof does not.
7. **Frontend proof stays in CI when frontend-related surfaces move.** If a change touches frontend code or backend/API/auth surfaces that materially affect the web app, PR and mainline must include frontend tests; backend-heavy diffs do not get to skip frontend proof when the user-facing surface is coupled.
8. **Frontend proof means real `apps/web` tests, not local-only folklore.** The target CI contract is `apps/web` test ownership in GitHub for both PR and mainline when triggered, including the deterministic web test suites the repo already owns. Build-only or route-invariant checks are not enough by themselves.
9. **Mainline backend proof should be broad by default, not hand-picked clinic theater.** The authoritative mainline lane should run all deterministic backend unit/integration/E2E suites that are practical in CI across Grove, platform-core, API, temporal-worker, and deterministic solution suites, while leaving mislocated solution-owned Temporal worker proofs manual-only instead of silently treating them as generic platform coverage.
10. **Simplification means deleting duplicate topology.** No second live PR workflow, no temporary “transition” PR `k3d` lane that never dies, and no docs that claim the same suite is authoritatively owned by both PR and mainline.

## Target Coverage Matrix

| Test surface | PR target | Mainline target |
| --- | --- | --- |
| Docs / contracts / static | yes | yes |
| Python lint / types | yes | yes |
| Backend unit tests | yes | yes |
| Backend integration tests | yes, if deterministic | yes |
| Backend non-runtime E2E | yes, if deterministic and runnable on the runner shell | yes |
| Frontend unit/integration | yes when frontend/coupled | yes when frontend/coupled |
| Frontend Playwright/browser | yes when frontend/coupled | yes when frontend/coupled |
| Compose runtime smoke | no | no |
| `k3d` runtime proof | no | yes, one authoritative run |
| Traceability harness | no | yes, inside mainline full run |
| Checklist-derived `k3d` subset | no | yes, inside mainline full run |
| Live providers / real LLM | no | no |
| Voice / SIP / telephony live | no | no |

Interpretation:

1. PRs stay shell-only. They do not retain Docker Compose runtime smoke as a “cheap reassurance” lane.
2. Mainline owns the only authoritative cluster-backed runtime truth, using one `k3d` bootstrap per run.
3. Mainline may still execute deterministic shell tests in the same workflow run, but cluster-backed proof is owned there and nowhere else.

## Non-Goals

- No persistent shared `k3d` cluster across different workflow runs, events, or SHAs.
- No second live PR workflow or PR-only replacement lane disguised as “temporary”.
- No PR-only runtime smoke lane retained as “cheap reassurance.”
- No broad CI feature land grab beyond the topology simplification and ownership cleanup described here.
